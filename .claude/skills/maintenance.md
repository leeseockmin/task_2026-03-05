# /maintenance — 코드 유지보수

`developer-a` 또는 `developer-b` 에이전트로서 기존 코드를 유지보수합니다.

## 유지보수 대상

$ARGUMENTS

---

## 작업 전 확인 사항

1. `.claude/skills/team-rules.md` 파일을 읽고 **팀 개발 룰** 숙지
2. 변경 대상 파일 전체를 읽고 영향 범위 파악
3. Breaking change 발생 가능성 → 시니어 개발자에게 먼저 보고

---

## 작업 유형 판단

요청 내용을 분석하여 아래 유형 중 해당하는 작업을 수행합니다.

### 타입 A: 버그 수정
1. 버그 발생 파일 읽기 및 원인 분석
2. 원인 설명 후 최소한의 수정으로 해결
3. 관련 테스트 파일 확인 및 업데이트
4. 수정 완료 후 QA에게 회귀 테스트 요청

### 타입 B: 기능 개선 (기존 기능 변경)
1. 변경 전/후 동작 명확히 정의
2. Command 측 변경 시 — EF Core 사용, 마이그레이션 필요 여부 확인
3. Query 측 변경 시 — Dapper SQL 파라미터 바인딩 유지
4. 컨트롤러 시그니처 변경 시 — API 버전 호환성 시니어에게 확인

### 타입 C: 성능 최적화
1. 현재 쿼리/로직 파악
2. 성능 문제 원인 분석 및 설명
3. 최적화된 Dapper 쿼리 작성 또는 EF Core Include 개선
4. 인덱스 추가가 필요한 경우 → 시니어 개발자에게 MySQL 인덱스 설계 요청

### 타입 D: 리팩토링
1. 리팩토링 전 코드 전체 파악
2. 기존 동작을 유지하면서 코드 개선
3. 관련 없는 영역까지 수정 금지 (범위 최소화)
4. 완료 후 QA에게 회귀 테스트 요청

---

## 유지보수 로깅 기준

- 수정한 로직에서 기존 로깅이 올바른지 확인
- 애플리케이션 로그 → `ILogger<T>` (Serilog)
- 서비스 이벤트 로그 → `IMongoDbServiceLogRepository` (MongoDB)
- 수정 과정에서 `Console.WriteLine` 절대 추가 금지

---

## 완료 후 처리

1. 변경한 파일 목록과 변경 내용 요약 출력
2. `/test`로 QA에게 회귀 테스트 요청
3. `/code-review`로 시니어 개발자에게 리뷰 요청
