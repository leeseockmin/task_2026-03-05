# /test — 테스트 작성 및 실행

`qa-engineer` 에이전트로서 아래 대상에 대한 테스트를 작성하고 실행합니다.

## 테스트 대상

$ARGUMENTS

---

## 테스트 진행 단계

### 1단계: 대상 코드 파악
- 테스트 대상 파일(핸들러, Repository, 컨트롤러) 읽기
- `BackEnd.Tests/` 내 기존 테스트 파일 존재 여부 확인
- 테스트해야 할 시나리오 목록 식별

### 2단계: 단위 테스트 작성 (xUnit + Moq)

**Command 핸들러 테스트 시나리오:**
- 정상 입력 → 성공 반환
- 유효하지 않은 입력 → 예외 발생
- Repository 호출 횟수 검증
- MongoDB 서비스 로그 호출 검증 (`IMongoDbServiceLogRepository`)

**Query 핸들러 테스트 시나리오:**
- 존재하는 데이터 조회 → DTO 반환
- 존재하지 않는 데이터 조회 → null 반환
- 조회 실패 시 경고 로그 발생 여부

**테스트 명명 규칙:**
```
{메서드명}_{시나리오}_{예상결과}
예: Handle_유효한_요청_시_사용자ID를_반환해야_한다
```

### 3단계: 통합 테스트 작성 (API 엔드포인트)

각 엔드포인트별 검증:
- `POST` → 201 Created (정상) / 400 Bad Request (잘못된 요청)
- `GET /{id}` → 200 OK (존재) / 404 Not Found (미존재)
- `GET` (목록) → 200 OK
- `PUT /{id}` → 200 OK / 400 / 404
- `DELETE /{id}` → 204 No Content / 404

### 4단계: 테스트 실행

```bash
# 전체 테스트 실행
dotnet test

# 특정 클래스 실행
dotnet test --filter "FullyQualifiedName~{테스트클래스명}"

# 커버리지 포함 실행
dotnet test --collect:"XPlat Code Coverage"
```

### 5단계: 결과 보고

```
## 테스트 결과: {기능명}

### 단위 테스트
- 전체: N개 | 통과: N개 | 실패: N개

### 통합 테스트
- 전체: N개 | 통과: N개 | 실패: N개

### 실패한 테스트
- {테스트명}: {실패 원인}

### 버그 발견 시
{표준 버그 보고 형식으로 작성}
```

## 버그 수정 요청 규칙

> **QA는 코드를 직접 수정하지 않습니다.**

버그가 발견되면 아래 절차를 따릅니다.

1. 버그 보고서 작성 (파일, 라인, 원인, 재현 방법)
2. `developer-a` 또는 `developer-b` 중 **랜덤**으로 한 명을 선택하여 수정 요청
3. 수정 완료 후 회귀 테스트 재수행

**담당자 배정 기준 (랜덤 선택):**
```
홀수 버그 번호 → developer-a에게 수정 요청
짝수 버그 번호 → developer-b에게 수정 요청
```

**수정 요청 형식:**
```
[개발자A / 개발자B] 수정 요청

버그 번호: #N
파일: {파일경로}:{라인}
원인: {원인 설명}
수정 내용: {수정 방향}
```

## 테스트 체크리스트

### 정상 케이스
- [ ] 정상 입력 시 올바른 응답 코드와 데이터
- [ ] 서비스 이벤트 로그 (MongoDB) 호출 여부

### 예외 케이스
- [ ] 빈 값, null 값 처리
- [ ] 경계값 (최소/최대) 처리
- [ ] 존재하지 않는 리소스 → 404
- [ ] 잘못된 형식 → 400

### 보안 케이스
- [ ] SQL Injection 시도 차단 여부
- [ ] 인증 없이 보호 엔드포인트 접근 → 401

### 회귀 테스트
- [ ] 기존 기능이 변경 후에도 정상 동작
- [ ] 이전 버그 재발 없음
